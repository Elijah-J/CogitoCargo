# `Cargo.lock`: the reproducible-build record

## The Move

The first time you run `cargo build` (or `cargo run`, or `cargo check`)
inside a Cargo package, Cargo creates a new file at the package root —
next to `Cargo.toml` — named `Cargo.lock`. The lockfile records the
*exact* resolved version of every dependency the build picked up. On
later builds Cargo notices `Cargo.lock` exists and reuses the versions
it lists, even if newer matching versions are now available. That gives
the project a *reproducible build*: same source plus same lockfile
gives the same compiled artifact, today or six months from now. Cargo
manages the file; do not edit it by hand.

## Mental Model Delta

- *Before:* "Lesson 065 added `rand = "0.8.5"` to `[dependencies]` and
  Cargo fetched the actual version. I had no picture of how Cargo
  *records* what it picked, or how the next build reuses that record."
- *After:* "Cargo writes the picked versions into `Cargo.lock` after
  the first build. Future builds read that lockfile and reuse those
  exact versions, even if newer matching versions are out there. That
  is what makes Cargo builds reproducible. Even a no-dependency package
  gets a (sparse) `Cargo.lock`. The file is Cargo's; my job is to leave
  it alone."

## Try It

In a directory you can write to:

```console
$ cargo new --vcs none lock_no_deps
    Creating binary (application) `lock_no_deps` package
$ cd lock_no_deps
$ ls
Cargo.toml  src
```

Notice: no `Cargo.lock` yet. `cargo new` does *not* create the
lockfile. Now build:

```console
$ cargo build
   Compiling lock_no_deps v0.1.0 (/path/to/lock_no_deps)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 1.67s
$ ls
Cargo.lock  Cargo.toml  src  target
```

`Cargo.lock` appeared, alongside the familiar `target/` from lesson
064. Look at its contents:

```console
$ cat Cargo.lock
# This file is automatically @generated by Cargo.
# It is not intended for manual editing.
version = 4

[[package]]
name = "lock_no_deps"
version = "0.1.0"
```

Two things to read off this file:

1. The header comment is Cargo telling you, in plain text, *do not
   edit this file*. Cargo regenerates it.
2. The body is short ("a bit sparse," in the Book's words) because
   this package has no dependencies. There is one `[[package]]` entry
   — the package itself.

Now run `cargo build` a second time, no edits:

```console
$ cargo build
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.01s
```

Same lesson-064 cache-hit shape. Cargo saw the source and lockfile
were unchanged and did nothing. The lockfile, once written, is
reused.

To see the lockfile actually *recording dependencies*, repeat the
same recipe in a fresh package whose `Cargo.toml` lists
`rand = "0.8.5"` (lesson 065). Run `cargo build` there and `cat
Cargo.lock`. Instead of one `[[package]]` entry, you get one for
your package and one for each dependency Cargo resolved
(`rand`, `rand_chacha`, `rand_core`, `getrandom`, `libc`,
`cfg-if`, ...) — each with its exact picked version. The full
transcript is in the evidence appendix.

## What Changed

- `Cargo.lock` is the file Cargo writes at the package root after
  the first `cargo build` (or `cargo run`, or `cargo check`). It
  records the *exact* resolved version of every dependency.
- Even a no-dependency package gets a (sparse) lockfile with one
  `[[package]]` entry — the package itself.
- On the next build, Cargo reads `Cargo.lock` and reuses those
  versions, even if newer matching versions are available on
  crates.io. The Book's framing: "your project will remain at
  0.8.5 until you explicitly upgrade, thanks to the *Cargo.lock*
  file." (The Book uses 0.8.5 as its running example; the host
  probe locked `rand v0.8.6` because `"0.8.5"` in `Cargo.toml` is
  SemVer-caret-compatible with later 0.8.x releases — the rule
  lesson 065 deferred. The reproducibility claim is unchanged:
  whatever version Cargo picked is what future builds reuse.)
- Reproducibility rule: same source plus same `Cargo.lock` gives
  the same compiled artifact, regardless of when you build.
- Cargo manages the file. The first two lines say so. Do not edit
  by hand; let Cargo regenerate it when `Cargo.toml`'s
  `[dependencies]` change.
- For a binary application (the kind `cargo new` scaffolds today),
  the Book recommends checking `Cargo.lock` into source control —
  "it's often checked into source control with the rest of the code
  in your project."

## Check Yourself

You just ran `cargo new --vcs none playground` and `cd playground`,
with `Cargo.toml` left at its default (empty `[dependencies]`).

- After `cargo build`, what new file appears at the package root?
- The package has *no* dependencies. Will `Cargo.lock` still appear?
- Cargo offers no warning if you hand-edit `Cargo.lock`. Why is the
  Book's advice still "don't"?
- Same source code, lockfile lost. Cargo runs the next build. Will
  the resulting binary necessarily be byte-identical to one built
  yesterday with the lost lockfile?

(Answers: `Cargo.lock`. Yes — even no-dependency packages get a
sparse lockfile. Hand edits will be overwritten the next time
Cargo regenerates the file, and meanwhile they break the
reproducibility guarantee — Cargo wrote the exact resolved
versions; an edit no longer represents what Cargo would resolve.
And no — without the lockfile Cargo re-resolves from
`Cargo.toml`'s SemVer ranges, possibly picking newer matching
versions of dependencies; the lockfile is exactly the missing
half of the reproducibility guarantee.)

## What To Ignore For Now

- The internal structure of `Cargo.lock` (TOML format,
  `[[package]]` arrays, `source`, `checksum`, `dependencies`
  fields, the `version = 4` schema marker). Today: *Cargo records
  resolved versions in this file*. Schema details are deferred.
- `cargo update` — the explicit "upgrade my dependency versions"
  command. The Book introduces it on the next page (Ch2, lines
  608ff). Future move.
- Library-vs-binary lockfile-checkin: library crates published to
  crates.io often *do not* check `Cargo.lock` into source control.
  Today's scope is binary applications, the kind `cargo new`
  scaffolds.
- SemVer caret-range resolution rules (e.g., `"0.8.5"` matches
  `0.8.5..0.9.0`). Lesson 065 noted "your output may differ" and
  deferred this; still deferred.
- `--locked` and `--frozen` flags for CI: fail the build if Cargo
  would have to change `Cargo.lock`. Future move.
- Workspaces, nested packages, `[patch]` overrides, and lockfile
  format-version (`version = 4`) migrations.
