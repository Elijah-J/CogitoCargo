### Probe 1 — working (`demo.rs`, trait declaration `&Counter`, impl method `&Self`)

```rust
struct Counter {
    count: u32,
}

trait Combine {
    fn combine(&self, other: &Counter) -> u32;
}

impl Combine for Counter {
    fn combine(&self, other: &Self) -> u32 {
        self.count + other.count
    }
}

fn main() {
    let a = Counter { count: 7 };
    let b = Counter { count: 35 };
    let result = a.combine(&b);
    println!("result = {}", result);
    println!("b.count still = {}", b.count);
}
```

```
$ /Users/eli/.cargo/bin/rustc demo.rs
$ echo $?
0
$ ./demo
result = 42
b.count still = 35
$ echo $?
0
```

Compiles silently. Prints `result = 42` (7 + 35) and `b.count still =
35` (`b` is alive after the call — the lesson-113 "still owns"
property re-witnessed once, not centered). The output is the empirical
witness for "`&Self` in the non-receiver slot of the *impl method*
works inside `impl Combine for Counter`" — what 113 did with
`&Counter` in both the trait declaration and the impl signature, today
keeps `&Counter` in the trait declaration and substitutes `&Self` only
in the impl signature. Lesson 112's contract-matching rule is
satisfied after `Self` resolution: inside `impl Combine for Counter`,
the impl's `&Self` resolves to `&Counter`, matching the trait
declaration's `&Counter` exactly.

### Probe 4 — Check Yourself (a)/(b) ground (`tiny.rs`)

## Direct prerequisite summary

- **Lesson 113** (load-bearing): installed `&Type` as a non-receiver
  reference parameter on a trait method, with the `&value` call-site
  shape and the "caller still owns" property. Lesson 113 explicitly
  named today's move in its *What To Ignore For Now*: "**`&Self`
  vs `&Counter` inside the impl** — same meaning here (`Self` is the
  type alias from lesson 100). The rmp source uses both; today
  centers the named-type form because it is directly readable."
  Today picks up that exact deferral; the only delta is the spelling.
- **Lesson 100** (load-bearing): installed the rule "Within an `impl`
  block, the type `Self` is an alias for the type that the `impl`
  block is for" (Book ch05-03:62-63 verbatim, the same passage cited
  here). 100 used the alias in a return-type slot inside an *inherent*
  impl (`fn new() -> Self`); today extends the alias to a
  *parameter-type* slot inside a *trait* impl. The Reference
  `paths.md:316-322` covers both extensions: implementations include
  both inherent and trait impls, and the alias is allowed wherever
  the impl is allowed.
- **Lesson 111** (load-bearing): installed the
  `impl Trait for Type { ... }` scaffold. Today's working probe uses
  this exact scaffold; the Self-alias rule applies inside it the same
  way it does inside lesson 100's inherent `impl Type { ... }`
  scaffold per Reference `paths.md:316-322`.
- **Lesson 112** (load-bearing): installed the rule that the impl
  method signature must match the trait declaration exactly. Today's
  centered move (impl `&Self`, trait `&Counter`) only works because
  `Self` is the type alias from lesson 100: post-resolution, the
  impl's `&Counter` matches the trait's `&Counter`. Without lesson
  112, the post-resolution match would not be the relevant question.

## Deferrals

Each item below was named in the lesson's *What To Ignore For Now*
and is not load-bearing for today's claims:

- `&mut Self` in a non-receiver slot (composes 101 with today).
- `Self` as a return type in a trait method (`Self: Sized` and
  dispatch).
- `Self::method()` self-references inside an impl body.
- `where Self: Bound` clauses.
- `Self` inside a *trait declaration* itself (Reference
  `paths.md:316-318` covers it; framing it from the declaration side
  is a separate move).
- Multiple `&Self` parameters in one signature; `&'a Self`; `&&Self`.
- The full borrow-checker rules, lifetimes, shared-vs-mut
  exclusivity — wholesale deferred from 113.
