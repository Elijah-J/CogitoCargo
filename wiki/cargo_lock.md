# `Cargo.lock`

`Cargo.lock` records Cargo's resolved package state. When a project has
dependencies, that includes the exact dependency versions Cargo selected. Cargo
manages this file.

## Where I saw it

The Rust Book says running `cargo build` for the first time creates
`Cargo.lock` at the package root:

```text
hello_cargo/
|-- Cargo.lock
|-- Cargo.toml
`-- src/
    `-- main.rs
```

The beginner `hello_cargo` project has no external dependencies, so its lock
file is not very interesting yet. It still shows that Cargo tracks exact
resolution state separately from `Cargo.toml`.

## Lockfile shape I saw

The generated `hello_cargo` lockfile currently looks like this:

```toml
# This file is automatically @generated by Cargo.
# It is not intended for manual editing.
version = 4

[[package]]
name = "hello_cargo"
version = "0.1.0"
```

The generated comment matches the main rule for now: do not edit this file by
hand.

The top-level `version = 4` belongs to the lockfile itself. It is not the same
thing as the package version.

`[[package]]` starts a package entry in the lockfile. In this tiny project, the
only package entry is `hello_cargo` itself:

```toml
name = "hello_cargo"
version = "0.1.0"
```

## `Cargo.lock` versus `Cargo.toml`

`Cargo.toml` is the file a developer edits. It describes the package and the
dependency requests.

`Cargo.lock` is the file Cargo maintains. When dependencies exist, it records
the exact dependency versions Cargo selected.

## Useful guardrail

Do not treat `Cargo.lock` as another place to configure dependencies. Edit
`Cargo.toml`; let Cargo update `Cargo.lock`.
